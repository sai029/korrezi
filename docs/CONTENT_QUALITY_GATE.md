# 記事品質ゲート（コンテンツ採点AI）方針

> 最終更新: 2026-06-27 ／ 関連: [`ARCHITECTURE.md`](ARCHITECTURE.md) ・ [`SPECIFICATION.md`](SPECIFICATION.md)

子ども向けニュースアプリに記事を取り込む際、**記事の質を AI で採点し、載せるかどうかを判断する**仕組み。
GNews から取得した記事を `news_pool` に書き込む前段に「ゲート」として挟む。

---

## 基本方針（重要）

- **カスタムモデルは作らない。** 本プロジェクトは既に Vertex AI の Gemini
  (`gemini-2.5-flash`) をプロンプトで呼ぶ構成が完成している（`functions/src/index.ts`）。
  採点も「既存と同じ Gemini 呼び出しに、採点用プロンプトを 1 つ追加する」だけで実現する。
  モデル訓練（教師データ作成・デプロイ・維持）はコストと開発停滞の割に利点がないため採用しない。
  → これは **LLM-as-a-Judge** パターン。
- **4 軸はまとめて 1 回の Gemini 呼び出しで採点する**（呼び出し回数＝コスト＝レイテンシのため分割しない）。
- **ゲートは「子ども向け変換（`toChildFriendly`）の前」に置く。** 落とす記事に変換コストを払わない。
  ```
  GNews取得 → 【採点ゲート(新規)】→ 合格分のみ → toChildFriendly → news_pool 書き込み
  ```
- **採点は再現性重視**: 既存の変換は `temperature: 0.7`（創作なので妥当）だが、
  採点は別設定で `temperature: 0` ＋ `responseSchema`（構造化出力）で JSON 形を強制し、ブレを抑える。
- **Vertex の `safetySettings` も併用**（暴力・性的表現等の閾値）してプロンプトと二重に防御する。

---

## 4 つの評価軸

AI に記事テキストを入力し、各軸をスコアリング（1〜5）または判定させる。

| 軸 | 内容 | 高スコア例 | 低スコア / 除外例 |
|---|---|---|---|
| ① 教育的価値・テーマ適正（最重要） | 視野を広げ、中学受験や現代社会の理解に繋がるか | SDGs・科学・技術・国際情勢・環境・経済の仕組み・文化歴史・社会課題の解決事例 | 芸能スキャンダル・ゴシップ・凄惨な事件事故・単なる流行紹介・政治的プロパガンダ |
| ② 思考のフック（背景・理由の記述） | 「なぜ起きたか（背景）」「これからどうなるか（影響・問い）」があるか | 背景と影響に触れている | 「〇〇が起きました」だけの 1 行ニュース・結論のみでプロセスが無い |
| ③ 信頼性・客観性 | 親が安心して見せられるか | 事実と意見が区別／煽り表現が無い／複数視点（メリ・デメ）に触れる | 感情的・過激な煽りタイトル、片側のみの主張 |
| ④ セーフティ（安全面） | 子どもへの心理的安全 | — | 残虐・性的・差別・自殺/自傷、過度に恐怖を煽る天災描写 → **即除外** |

> ②は GNews 無料枠の短い `description`／途中で切れる `content` からは判定が甘くなりがち。
> この軸は「判定不能（unknown）」を許容する設計にする。①④は短文でも十分判定可能。

---

## 段階導入: ハイブリッド方式（採用）

「記録のみ」と「自動除外」で **AI の採点処理は同一**。違いは *スコアをどう使うか* だけで、
自動除外が「精度が悪い」わけではない。リスクの正体は **閾値が適切かどうか**。

そこで軸ごとに分けて運用する:

- **④ 安全性 → 最初から自動除外。**
  判定が高確度かつ、子どもに一度でも見せてはいけないため、記録のみで待つ意味がない。
- **①②③ 品質軸 → 当面は「記録のみ」（全記事を載せつつスコアと判定理由を記録）→ 閾値が固まり次第、自動除外へ移行。**
  閾値調整には実データの採点実績が要る。特に②は短文で甘くなりがちで、いきなり自動除外すると
  良記事を誤って落とす恐れがある。

### 自動除外へ移行する条件

- 移行の前提は「**閾値に確信が持てること**」（AI が賢くなるのを待つのではない）。
- GNews は 1 日約 10 件。閾値を自信を持って決めるには **採点実績 100〜200 件程度**が目安。
  - 毎日の自動取得のみ → 約 2〜3 週間
  - 手動で `fetchNews` を多めに回せば数日に短縮可能
- 記録された各軸スコアを確認し、良記事／悪記事がどのスコアで分かれるかを見て閾値を決める。

## データ記録（Firestore）

採点結果は合格記事の `news_pool/{id}` に **`quality_review` 入れ子マップ**としてまとめて格納する
（表示用フィールドはフラットなまま。責務はコード側 `scoreArticle` とデータ側マップの両方で分離）。

```jsonc
"quality_review": {
  "verdict": "approved",              // 現状 news_pool に載るのは合格のみ
  "safety": { "passed": true, "flagged": [] },  // ④安全性。NG なら除外され doc ごと存在しない
  "scores": {                          // ①②③（1〜5 / 判定不能は null）
    "educationalValue": 4,
    "thinkingHook": null,              // 短文で判定不能のとき null
    "reliability": 3
  },
  "reason": "環境問題の背景に触れた教育的な記事。",
  "model": "gemini-2.5-flash",
  "schema_version": 1,
  "scored_at": "<Timestamp>"
}
```

- **安全 NG（④）の記事は news_pool に書き込まれない**ため、現状「落とした記事」のスコアは残らない。
  品質軸は記録のみなので落ちるのは安全 NG の少数のみ。将来それも監査したくなったら専用コレクションを足す。
- クライアントは `quality_review` を読まなくてよい（未知フィールドは Dart モデルが無視する）。
- 実装: `functions/src/index.ts` の `scoreArticle`（採点）→ `ingestArticles`（ゲート→変換→書込）。
  採点モデルは `temperature: 0` ＋ `responseSchema` ＋ `safetySettings`（BLOCK_LOW_AND_ABOVE）。

### 採点失敗時の挙動（fail-closed）

採点呼び出しが失敗した（＝安全性を確認できない）記事は **除外**する。Vertex の安全フィルタが
応答自体をブロックした場合も応答が空になり、同様に除外される。Gemini 障害時はフィードが空に
なりうるが、未検証の記事を子どもに見せるよりは安全と判断した。
